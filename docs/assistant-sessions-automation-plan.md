# Assistant Sessions Script Transaction Plan

This document defines a production-ready plan for evolving Rewrite sessions into a generic assistant session system.

The core idea:

1. The assistant can inspect context with read APIs.
2. The assistant can produce JavaScript that transforms that context.
3. The script records proposed writes into a transaction.
4. The app dry-runs the transaction and shows a preview.
5. The user approves once.
6. The app applies the transaction.

This gives the assistant a powerful generic orchestration layer while keeping durable writes behind preview and approval.

## Goals

- Keep the user-facing model simple: a session is a chat-like workspace where the assistant helps complete a task.
- Evolve Rewrite sessions into the generic assistant session model.
- Let the assistant author JavaScript for data shaping, filtering, mapping, batching, and conditional logic.
- Expose app context through typed capabilities instead of direct store access.
- Capture durable writes as transaction operations.
- Require dry-run preview and user approval before applying writes.
- Preserve existing Rewrite behavior during migration.
- Keep domain behavior in capability APIs and tools while the session model stays generic.

## Current State

Rewrite sessions currently provide:

- Local session persistence through `localforage` in `RewriteService`.
- Session messages with `system`, `user`, `assistant`, and `tool` roles.
- Structured assistant responses for rewrite results.
- Tool definition injection and bounded tool execution.
- Diff and apply behavior for editable fields.
- Session UI with history, session list, tool toggles, regeneration, edit, delete, and abort.

Relevant files:

- `src/extensions/built-in/rewrite/RewriteService.ts`
- `src/extensions/built-in/rewrite/composables/useRewriteSessions.ts`

## Core Design

Use one generalized session model. Rewrite-specific behavior remains optional session data. Generic automation is represented as script output plus a transaction.

```ts
export interface AssistantSession {
  id: string;
  templateId?: string;
  recipeId?: string;
  identifier: string;
  title?: string;
  createdAt: number;
  updatedAt: number;
  status?: AssistantSessionStatus;
  systemPrompt: string;
  messages: AssistantSessionMessage[];

  initialFields?: RewriteField[];

  scriptRun?: AssistantScriptRun;
  transaction?: AssistantTransaction;

  allowedCapabilities?: string[];
  disabledCapabilities?: string[];
  metadata?: Record<string, unknown>;
}

export type AssistantSessionStatus =
  | 'idle'
  | 'running'
  | 'script_ready'
  | 'preview_ready'
  | 'applying'
  | 'completed'
  | 'failed'
  | 'aborted';
```

`recipeId` and `templateId` are descriptive and optional. Runtime behavior comes from capabilities, scripts, and transaction operations.

## Script Runs

An assistant script is JavaScript generated by the LLM and executed by a restricted script runner.

```ts
export interface AssistantScriptRun {
  id: string;
  status: 'draft' | 'validated' | 'executed' | 'failed';
  code: string;
  result?: AssistantScriptResult;
  error?: string;
  createdAt: number;
  updatedAt: number;
}

export interface AssistantScriptResult {
  summary?: string;
  logs: string[];
  transaction?: AssistantTransaction;
  data?: unknown;
}
```

The script runner provides a controlled context object:

```ts
export interface AssistantScriptContext {
  read: <T = unknown>(capabilityName: string, args?: unknown) => Promise<T>;
  tx: AssistantTransactionBuilder;
  template: {
    render: (template: string, data: unknown) => string;
  };
  log: (...values: unknown[]) => void;
}

export interface AssistantTransactionBuilder {
  add: (operation: {
    capabilityName: string;
    args: unknown;
    label?: string;
  }) => void;
}
```

The assistant can use normal JavaScript control flow:

- Filtering.
- Sorting.
- Grouping.
- Mapping.
- Conditional field inclusion.
- Batching.
- Calling read capabilities.
- Rendering Handlebars templates.
- Recording writes through `ctx.tx.add`.

## Script Runner Requirements

The script runtime must be isolated from the browser and app internals.

Required controls:

- No DOM access.
- No `window`, `document`, `localStorage`, `indexedDB`, or direct Pinia store access.
- No network APIs.
- No dynamic imports.
- No timers except runner-owned timeout enforcement.
- No access to extension APIs except the provided `ctx` object.
- Execution timeout.
- Memory/output limits.
- Maximum read calls.
- Maximum transaction operations.
- Captured logs with truncation.
- Deterministic transaction ids and operation ids assigned by app code.

Implementation options:

- Use a restricted JavaScript interpreter.
- Use an isolated worker with a locked-down global object.
- Use a library such as SES or a small JS interpreter if it fits bundle/runtime constraints.

The runner should be replaceable behind an interface:

```ts
export interface AssistantScriptRunner {
  validate(code: string): Promise<ScriptValidationResult>;
  execute(code: string, context: AssistantScriptContext): Promise<AssistantScriptResult>;
}
```

## Transactions

A transaction is the generic representation of proposed writes.

```ts
export interface AssistantTransaction {
  id: string;
  status: 'draft' | 'previewed' | 'approved' | 'applying' | 'applied' | 'failed';
  operations: AssistantOperation[];
  preview?: AssistantTransactionPreview;
  result?: AssistantTransactionResult;
  createdAt: number;
  updatedAt: number;
}

export interface AssistantOperation {
  id: string;
  capabilityName: string;
  args: unknown;
  label?: string;
}
```

Scripts record write operations through `ctx.tx.add`. The app owns transaction ids, operation ids, validation, dry-run preview, approval, and apply.

Transaction preview:

```ts
export interface AssistantTransactionPreview {
  operationPreviews: AssistantOperationPreview[];
  summary: {
    operations: number;
    warnings: number;
    errors: number;
  };
  canApply: boolean;
}

export interface AssistantOperationPreview {
  operationId: string;
  capabilityName: string;
  label?: string;
  summary: string;
  details?: unknown;
  diff?: unknown;
  warnings?: string[];
  errors?: string[];
}
```

Transaction result:

```ts
export interface AssistantTransactionResult {
  operationResults: AssistantOperationResult[];
  summary: {
    applied: number;
    failed: number;
    skipped: number;
  };
}

export interface AssistantOperationResult {
  operationId: string;
  capabilityName: string;
  status: 'applied' | 'failed' | 'skipped';
  summary: string;
  details?: unknown;
  error?: string;
}
```

The preview and result structures are generic. Domain-specific data lives inside `details` or `diff`, and is produced by the capability implementation.

## Capability Model

Capabilities are the script-facing API surface. They can be backed by existing tools, services, or stores, but scripts only see the capability interface.

```ts
export type CapabilityMode = 'read' | 'write';

export interface AssistantCapability {
  name: string;
  displayName?: string;
  description: string;
  namespace?: string;
  mode: CapabilityMode;
  parameters: Record<string, unknown> | object;

  action: (args: unknown) => Promise<unknown> | unknown;
  dryRun?: (args: unknown) => Promise<CapabilityPreview> | CapabilityPreview;
}

export interface CapabilityPreview {
  summary: string;
  details?: unknown;
  diff?: unknown;
  warnings?: string[];
  errors?: string[];
  canApply?: boolean;
}
```

Rules:

- Read capabilities are available through `ctx.read`.
- Write capabilities are available through `ctx.tx.add`.
- Write capabilities implement `dryRun`.
- Write capability `action` runs during approved transaction apply.
- Capability args are schema-validated before dry-run and apply.

Existing `ToolDefinition` can be adapted into `AssistantCapability` to avoid duplicating registrations.

Recipes are reusable assistant instructions and default capability scopes.

export interface AssistantRecipe {

Examples:

- `core.rewrite-field`

- `core.characters-to-world-info`

The runner treats recipes as prompt and capability configuration.

## Character to Lorebook Workflow

The assistant inspects data with read capabilities, writes JavaScript to transform that data, and records world-info writes into a transaction. The app dry-runs those writes and asks the user for one final approval.

Recommended capabilities:

- `character.list` - read
- `character.get` - read
- `character.getMany` - read
- `worldInfo.book.list` - read
- `worldInfo.book.get` - read
- `worldInfo.book.ensure` - write
- `worldInfo.entry.list` - read
- `worldInfo.entry.createMany` - write
- `worldInfo.entry.updateMany` - write

Preferred technical flow:

### 1. User starts the session

The user sends a normal session message:

```json
{
  "id": "msg-user-1",
  "role": "user",
  "content": "Create a lorebook named \"Main Cast\" from all characters tagged \"main\". For each entry include description and personality if they exist.",
  "timestamp": 1760000000000
}
```

The session runner appends this to `session.messages`, sets `session.status = 'running'`, and calls the LLM with:

- Session system prompt.
- Current conversation messages.
- Capability documentation.
- Script context documentation.
- Structured response schema supporting `justification`, `toolCalls`, and `scriptProposal`.

### 2. LLM proposes a script

The assistant can propose JavaScript directly:

```json
{
  "justification": "I will gather the matching characters, render lorebook entries with templates, and record the required world-info writes as a transaction.",
  "scriptProposal": {
    "language": "javascript",
    "summary": "Create Main Cast lorebook entries from main-tagged characters.",
    "code": "async function main(ctx) {\n  const page = await ctx.read('character.list', {\n    offset: 0,\n    limit: 100,\n    tags: ['main'],\n    fields: ['id', 'name']\n  });\n\n  const ids = page.items.map((item) => item.id);\n  const characters = await ctx.read('character.getMany', {\n    ids,\n    fields: ['id', 'name', 'description', 'personality']\n  });\n\n  const entries = characters.items.map((character) => ({\n    key: [ctx.template.render('{{name}}', character)],\n    comment: ctx.template.render('{{name}}', character),\n    content: ctx.template.render(\n      '{{name}}\\n\\n{{#if description}}Description: {{description}}\\n{{/if}}{{#if personality}}Personality: {{personality}}\\n{{/if}}',\n      character\n    ),\n    sourceCharacterId: character.id\n  }));\n\n  ctx.tx.add({\n    capabilityName: 'worldInfo.book.ensure',\n    label: 'Ensure lorebook exists',\n    args: { name: 'Main Cast' }\n  });\n\n  ctx.tx.add({\n    capabilityName: 'worldInfo.entry.createMany',\n    label: 'Create entries from main characters',\n    args: {\n      bookName: 'Main Cast',\n      entries,\n      conflictStrategy: 'skip'\n    }\n  });\n\n  ctx.log(`Prepared ${entries.length} entries.`);\n  return { entryCount: entries.length };\n}"
  }
}
```

The script has freedom to shape data, but writes are only recorded through `ctx.tx.add`.

## UI Plan

Keep the current Rewrite session UI shape and generalize it incrementally.

Required areas:

- Session list sidebar.
- Main chat timeline.
- Script review panel.
- Script execution logs.
- Capability menu grouped by namespace and mode.
- Transaction preview panel when `session.transaction.status === 'previewed'`.
- Final approval controls: `Approve`, `Reject`, and `Ask to revise`.
- Transaction result panel after apply.

Rewrite-specific UI remains:

- Diff button.
- Apply button for rewrite field changes.
- Field change cards.

Generic additions:

- Code viewer for proposed script.
- Script validation and execution status.
- Operation preview list.
- Generic details renderer for JSON-like preview data.
- Optional table renderer when `details.rows` is present.
- Warning and error summaries.
- Status chip for `running`, `script_ready`, `preview_ready`, `applying`, `failed`, and `completed`.

Capability menu requirements:

- Group capabilities by `namespace`.
- Show `read` or `write` mode.
- Make write capabilities visually distinct.
- Explain that write capabilities are transaction-only.
- Keep session-level enable/disable independent from global settings.

## Storage and Migration

Keep storage migration additive in the first phase.

Phase 1 should keep the existing `rewrite-sessions` store and add optional fields to the stored shape.

Phase 2 can introduce an `assistant-sessions` store with read-through compatibility:

- Read existing rewrite sessions.
- Convert missing optional fields in memory.
- Save new or modified sessions in the new shape.
- Keep legacy sessions loadable.

Recommended compatibility rule:

```ts
function normalizeSession(session: RewriteSession | AssistantSession): AssistantSession {
  return {
    status: 'idle',
    allowedCapabilities: undefined,
    disabledCapabilities: [],
    metadata: {},
    ...session,
  };
}
```

## Implementation Phases

### Phase 1: Extend Existing Rewrite Sessions

- Add optional `status`, `scriptRun`, `transaction`, `allowedCapabilities`, `disabledCapabilities`, and `metadata` fields to the session type.
- Keep existing `RewriteSession` name initially to reduce churn.
- Add normalization when loading sessions.
- Add capability registration types and adapt existing tools into read/write capabilities.
- Keep existing rewrite diff/apply behavior unchanged.

Acceptance criteria:

- Existing rewrite sessions load and behave exactly as before.
- New optional fields persist while existing sessions stay loadable.
- Capability menu can display read/write mode.
- TypeScript remains strict without new broad `any` usage.

### Phase 2: Add Script Runner

- Add script proposal parsing to the session response schema.
- Add script validation.
- Add restricted script execution.
- Provide `ctx.read`, `ctx.tx.add`, `ctx.template.render`, and `ctx.log`.
- Generate app-owned script run, transaction, and operation ids.
- Add configurable limits:
  - max script runtime
  - max read calls
  - max logs
  - max proposed write operations
  - max serialized result size

Acceptance criteria:

- Scripts can call read capabilities.
- Scripts can render Handlebars templates.
- Scripts can record transaction operations.
- Scripts cannot access app internals outside provided capabilities.
- Script failures are stored on `session.scriptRun`.
- Aborting generation or script execution updates session status.

### Phase 3: Add Transaction Runner

- Validate transaction operations against registered capabilities.
- Run `dryRun` for every write operation.
- Store preview on `session.transaction`.
- Apply write actions after user approval.
- Store operation-level results.

Acceptance criteria:

- Valid script-created transactions produce previews.
- Invalid operations are reported to the session.
- Approved transactions apply in order.
- Failed operations are visible and recoverable.

### Phase 4: Add Core Data Capabilities

- Add read capabilities for characters and world info.
- Add write capabilities with required `dryRun`.
- Keep domain-specific preview data inside capability-owned `details` or `diff`.
- Ensure write capabilities validate inputs and report partial failures.

Acceptance criteria:

- Assistant scripts can list characters with pagination.
- Assistant scripts can inspect characters by id.
- Assistant scripts can list world info books and entries.
- Assistant scripts can create generic `worldInfo.entry.createMany` transactions.
- Dry-run preview works without mutating world info.

### Phase 5: Add Script and Transaction UI

- Render proposed scripts in the session view.
- Render script logs and errors.
- Render transaction preview in the session view.
- Add Approve, Reject, and Ask to revise controls.
- Execute approved transaction operations through the transaction runner.
- Store transaction results.

Acceptance criteria:

## Risks and Mitigations

Risk: Script execution exposes app internals.

Mitigation: Execute scripts in a restricted runner with only the `ctx` object exposed.

Risk: LLM writes mutate user data unexpectedly.

Mitigation: Writes are recorded as transaction operations, dry-run previews are generated first, and write actions run after user approval.

Risk: Scripts become expensive or unresponsive.

Mitigation: Enforce runtime, read-call, log, operation, and serialized-output limits.

Risk: Generated scripts are hard for users to understand.

Mitigation: Show script summary, code, logs, and transaction preview. The approval decision is based on the preview, with code available for inspection.

Risk: Existing Rewrite code becomes unstable during extraction.

Mitigation: Add optional fields first, then add the script runner and transaction runner after compatibility tests exist.

Risk: Capability previews become too large for session history.

Mitigation: Store compact summaries in messages and full preview data on `session.transaction.preview`.

Risk: Recipe behavior spreads into the session runner.

Mitigation: Treat recipes as prompt and capability configuration. Keep execution behavior in capabilities, scripts, and the transaction runner.
